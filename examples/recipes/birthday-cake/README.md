# Birthday Cake (Studio)

A **stylised birthday cake** — round board, two icing tiers, colored frosting
drips, a candle with flame, and scattered rainbow sprinkles — rendered under
soft studio lighting. The combined `scene.obj` (46,448 verts) carries **16
`usemtl` groups** in fixed order, so every material binds by 1-based
`group_index`:

| group_index | group           | role                    | material              |
|-------------|-----------------|-------------------------|-----------------------|
| 1           | `mat_plate`     | round cake board        | near-white glossy     |
| 2           | `mat_icing_lower` | lower icing tier      | pink glossy           |
| 3           | `mat_icing_upper` | upper icing tier      | light-pink glossy     |
| 4           | `mat_drip_a`    | drips on lower rim      | mint glossy           |
| 5           | `mat_drip_b`    | drips on lower rim      | yellow glossy         |
| 6           | `mat_drip_c`    | drips on upper rim      | blue glossy           |
| 7           | `mat_candle`    | candle body             | cream diffuse         |
| 8           | `mat_flame`     | candle flame            | orange glossy         |
| 9–16        | `mat_sprinkle1..8` | rainbow sprinkles    | 8 distinct glossy     |

The Octane render target exposes a single mesh pin, so a multi-part hero shot
must live in one combined OBJ with one group per material — `assign_material`
with `group_index` is the canonical way to bind them.

## Run

```bash
hermes mcp call octanex octane_queue_recipe --slug birthday-cake
```

## Notes

- The OBJ is generated by `scripts/gen_birthday_cake.py` (current: round board
  cylinder, hanging teardrop drips, rod sprinkles, thick candle, narrow teardrop
  flame). An earlier version concatenated groups **without rebasing face indices**
  (max face index 2305 vs 75,451 vertices), so almost all geometry was
  unreferenced — fixed; the committed `scene.obj` is clean (every face index in
  `[1, 46448]`).
- `octane-preview.png` is the native Octane render from the v2.1 realism pass
  (round board / drips / rod sprinkles confirmed live). **v2.2** (thicker candle +
  narrow tall flame + satin icing roughness 0.62/specular 0.2 + pulled-back camera
  `[4.5,3.5,5.1]`) is queued and awaiting a cold Octane restart + one-shot drain.
- The bridge's `create_material` only honors `diffuse/glossy/specular/metallic`
  (no `emission`), so the flame is a bright glossy orange, not a true emitter.
- Regenerate the asset: `uv run python scripts/gen_birthday_cake.py` (writes into
  the Octane container `assets/`), then re-queue the recipe.
- **Queue pipeline is single-source:** `scripts/queue_birthday_cake_v2.py` reads
  `materials` + `camera` from `scene.json` (no hardcoded `MATS`), so a camera or
  material edit in `scene.json` is what actually renders.

## Known issues / operational notes

- **Persistent bridge silent-exit risk.** Repeated import→render cycles on the
  persistent bridge in one session have been observed to wedge the render engine
  and then crash the bridge silently (final frame came out a plain blue field with
  no geometry, and the bridge window died with no error surfaced to `bridge.log`
  beyond the last `save attempt ... ok=true`). Recovery is a **full Octane X
  restart** (quit + reopen), then a single one-shot drain of a fresh queue. The
  bridge should guard `wait_for_render_ready` / the save path so an engine-busy or
  empty-mesh state is reported instead of crashing. (Logged 2026-07-12.)
- When iterating, prefer a **cold Octane** (File > New / quit+reopen) between
  major geometry regenerations rather than re-importing over a long-lived scene,
  to avoid duplicate/stale mesh state.
